Symfony - Tests unitaires et fonctionnels

Objectifs pédagogiques

A la fin de cette séance, vous serez en mesure de :

  • mettre en œuvre le framework de tests PHPUnit dans un projet Symfony
  • choisir le bon type de test (unitaire, intégration, fonctionnel, E2E)
  • écrire des tests lisibles et utiles (AAA, assertions pertinentes, isolation)

Introduction

Les tests sont là pour vérifier que ton application fait toujours ce qu'elle doit faire, même après des modifications. On parle de tests de non-régression.

Les grands types de tests

  • Test unitaire (UT) : teste une classe ou une fonction isolée, sans HTTP, sans base de données, sans container Symfony.
  • Test d'intégration : teste un service avec Symfony (Kernel + container), éventuellement avec une configuration particulière, mais sans scénario HTTP complet.
  • Test fonctionnel : teste un parcours applicatif via HTTP (routes, contrôleurs, formulaires, sécurité, templates).
  • Test E2E (end-to-end) : teste dans un vrai navigateur (JS, React, interactions), typiquement avec Panther. Plus lent, à réserver à quelques scénarios critiques.

Raccourci utile. UT = rapide et isolé. Intégration = container. Fonctionnel = HTTP. E2E = vrai navigateur.

Mise en place des tests (Symfony + PHPUnit)

Suis le tutoriel officiel Symfony pour initialiser phpunit et créer tes premiers tests :

https://symfony.com/doc/current/testing.html#application-tests

Ecrire des tests lisibles (AAA)

La structure la plus simple est AAA :

  • Arrange : je prépare les données
  • Act : j'appelle la méthode / l'action
  • Assert : je vérifie le résultat

Exemple de nommage : testTotalIsZeroWhenCartIsEmpty() (on décrit un comportement).

Tests unitaires (UT)

Un test unitaire ne dépend pas de Symfony. On teste une classe métier (service “pur”) en l'instanciant directement. Pas de Kernel, pas de base, pas de HTTP.

Exemple: calculateur de total

Voici un exemple minimal avec PHPUnit\Framework\TestCase. On suppose une classe CartTotalCalculator qui additionne les prix d'articles. 

<?php

namespace App\Tests\Unit;

use PHPUnit\Framework\TestCase;
use App\Service\CartTotalCalculator;

final class CartTotalCalculatorTest extends TestCase
{
    public function testTotalIsZeroWhenCartIsEmpty(): void
    {
        // Arrange
        $calculator = new CartTotalCalculator();

        // Act
        $total = $calculator->total([]);

        // Assert
        $this->assertSame(0, $total);
    }

    public function testTotalIsSumOfItemPrices(): void
    {
        // Arrange
        $calculator = new CartTotalCalculator();
        $items = [
            ['price' => 12],
            ['price' => 5],
        ];

        // Act
        $total = $calculator->total($items);

        // Assert
        $this->assertSame(17, $total);
    }
}

Objectif UT. Valider une règle métier rapidement, avec des entrées/sorties claires. Si tu as besoin du container Symfony, tu n'es plus en unitaire.

Tests d'intégration (KernelTestCase)

Quand tu veux tester un service “dans Symfony” (autowiring, paramètres, configuration), tu peux démarrer le Kernel et récupérer un service du container. Ça reste plus léger qu'un test fonctionnel (pas de HTTP). 

<?php

namespace App\Tests\Integration;

use Symfony\Bundle\FrameworkBundle\Test\KernelTestCase;
use App\Service\CartTotalCalculator;

final class CartTotalCalculatorIntegrationTest extends KernelTestCase
{
    public function testServiceIsWired(): void
    {
        self::bootKernel();
        $container = static::getContainer();

        $service = $container->get(CartTotalCalculator::class);

        $this->assertInstanceOf(CartTotalCalculator::class, $service);
    }
}

Tests fonctionnels (WebTestCase)

Client vs Crawler

Dans les tests fonctionnels, il faut bien différencier le client et le crawler.

Le client effectue les requêtes HTTP (GET/POST), clique sur des liens, soumet des formulaires. Le résultat est un crawler.

https://symfony.com/doc/current/components/browser_kit.html 

// Charger une nouvelle page
$crawler = $client->request('GET', '/clients');

// Cliquer sur un lien
$crawler = $client->clickLink('Details');

// Soumettre un formulaire
$crawler = $client->submitForm('Valider', [
    'login' => 'toto',
    'password' => '1234',
]);

Le crawler contient le HTML (ou du JSON) et permet de naviguer dans le DOM pour extraire des informations.

https://symfony.com/doc/current/components/dom_crawler.html 

// Extraire toutes les balises <p> du <body>
$dom = $crawler->filter('body > p');

// le premier
$dom = $crawler->filter('body > p')->first();
// le dernier
$dom = $crawler->filter('body > p')->last();

// récupère le contenu de l'élément sous forme de texte
$nom = $dom->text();

Assertions utiles

Les assertions valident ton test en indiquant des conditions de réussite.

Références : phpunit assertions, symfony response assertions.

  • assertResponseIsSuccessful() (statut 2xx)
  • assertResponseStatusCodeSame(401|403|404|500)
  • assertResponseRedirects('/login')
  • assertSelectorExists('h1')
  • assertSelectorTextContains('h1', 'Titre')

Exemple minimal

<?php

namespace App\Tests\Functional;

use Symfony\Bundle\FrameworkBundle\Test\WebTestCase;

final class HomePageTest extends WebTestCase
{
    public function testHomePageDisplaysTitle(): void
    {
        $client = static::createClient();
        $client->request('GET', '/');

        $this->assertResponseIsSuccessful();
        $this->assertSelectorExists('h1');
        $this->assertSelectorTextContains('h1', 'Village Green');
    }
}

Tester une page protégée (authentification)

Très souvent, une route est protégée par des rôles. Plutôt que de remplir le formulaire de login à chaque test, tu peux connecter un utilisateur directement. 

<?php

namespace App\Tests\Functional;

use Symfony\Bundle\FrameworkBundle\Test\WebTestCase;
use App\Entity\User;

final class AdminPageTest extends WebTestCase
{
    public function testAdminIsAccessibleForAdminUser(): void
    {
        $client = static::createClient();

        // Récupère un user en base (exemple) - à adapter à ton projet
        $user = static::getContainer()->get('doctrine')->getRepository(User::class)->findOneBy(['email' => 'admin@example.com']);

        $client->loginUser($user);

        $client->request('GET', '/admin');
        $this->assertResponseIsSuccessful();
    }

    public function testAdminIsForbiddenForAnonymous(): void
    {
        $client = static::createClient();
        $client->request('GET', '/admin');

        $this->assertResponseStatusCodeSame(302); // souvent redirection vers /login
        // ou: $this->assertResponseRedirects('/login');
    }
}

Base de données: isolation des tests

  • Utilise une base dédiée aux tests (env test).
  • Prépare des données reproductibles (fixtures, factory, jeux de données).
  • Assure-toi que chaque test démarre dans un état connu (reset DB, transactions, purge, etc.).

Panther (tests E2E avec vrai navigateur)

Les tests WebTestCase utilisent un navigateur “rudimentaire” : il n'exécute pas le JavaScript, donc il ne permet pas de tester correctement une application React.

Panther permet d'exécuter les tests dans un vrai navigateur (Chrome, Firefox) et de tester les interactions JS. Attention: c'est plus lent et plus fragile. Garde Panther pour quelques scénarios E2E vraiment importants (login, panier, paiement, etc.).

https://symfony.com/doc/current/testing/end_to_end.html
