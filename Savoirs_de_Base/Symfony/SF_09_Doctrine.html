DOCTRINE ORM - INTRODUCTION

Comprendre comment Symfony gère les données

(avec Doctrine ORM dans Symfony)

1. Pourquoi Doctrine ?

Dans une application web, les données :

  • ne viennent pas du contrôleur
  • ne viennent pas des vues
  • proviennent d’une base de données

Symfony s’appuie sur Doctrine ORM pour :

  • représenter les données sous forme d’objets PHP
  • éviter l’écriture directe de SQL dans l’application
  • structurer l’accès aux données de manière propre

Doctrine fait le lien entre le monde objet (PHP) et le monde relationnel (SQL).

2. Les quatre piliers de Doctrine

Pour comprendre Doctrine, il faut maîtriser quatre concepts essentiels :

  1. Entity
  2. Repository
  3. EntityManager
  4. DQL

3. L’Entity : représenter une table

Une Entity est une classe PHP qui représente :

  • une table de base de données
  • une ligne = un objet

3.1 Exemple simple d’Entity

namespace App\Entity;

use Doctrine\ORM\Mapping as ORM;

#[ORM\Entity]
class Article
{
    #[ORM\Id]
    #[ORM\GeneratedValue]
    #[ORM\Column]
    private ?int $id = null;

    #[ORM\Column(length: 255)]
    private string $title;

    public function getId(): ?int
    {
        return $this->id;
    }

    public function getTitle(): string
    {
        return $this->title;
    }
}

Une Entity :

  • ne contient pas de SQL
  • représente un concept métier
  • est indépendante de l’affichage

4. Le Repository : interroger les données

Chaque Entity possède un Repository.

Le Repository sert à :

  • récupérer des données
  • écrire des requêtes
  • centraliser l’accès à la base

4.1 Exemple de Repository

namespace App\Repository;

use App\Entity\Article;
use Doctrine\Bundle\DoctrineBundle\Repository\ServiceEntityRepository;

class ArticleRepository extends ServiceEntityRepository
{
    public function findPublished(): array
    {
        return $this->findBy(['published' => true]);
    }
}

Le Repository :

  • lit les données
  • ne fait pas d’affichage
  • ne contient pas de logique HTTP

5. L’EntityManager : le chef d’orchestre

L’EntityManager est le composant central de Doctrine.

Il est responsable de :

  • persister les objets
  • synchroniser avec la base
  • gérer les transactions

5.1 Exemple d’utilisation

$entityManager->persist($article);
$entityManager->flush();

Vous ne travaillez jamais directement avec la base Vous travaillez avec des objets

6. Lien entre Doctrine et les services

Doctrine est utilisé via des services.

Dans Symfony :

  • les Repositories sont des services
  • l’EntityManager est un service
  • vous les injectez par le constructeur

6.1 Injection dans un contrôleur

class ArticleController extends AbstractController
{
    public function __construct(
        private ArticleRepository $articleRepository
    ) {}

    public function index(): Response
    {
        $articles = $this->articleRepository->findAll();
        // ...
    }
}

Le contrôleur orchestre Doctrine travaille via ses services

7. Introduction au DQL (Doctrine Query Language)

7.1 Pourquoi le DQL ?

Doctrine ne vous oblige pas à écrire du SQL.

Il propose le DQL, un langage orienté objets, basé sur les Entities.

On ne requête pas des tables, mais des classes.

7.2 Exemple simple de DQL

$entityManager->createQuery(
    'SELECT a FROM App\Entity\Article a'
)->getResult();

Ce que cela signifie :

  • Article est une Entity
  • a est un alias
  • Doctrine génère le SQL automatiquement

7.3 DQL vs SQL (différence clé)

SQL DQL
Tables Entities
Colonnes Propriétés
Résultats bruts Objets PHP

Le DQL raisonne objet, pas relationnel.

8. DQL dans un Repository (bonne pratique)

Le DQL doit être écrit dans les Repositories, jamais dans les contrôleurs. 

public function findRecent(): array
{
    return $this->createQueryBuilder('a')
        ->orderBy('a.id', 'DESC')
        ->setMaxResults(5)
        ->getQuery()
        ->getResult();
}

Le QueryBuilder :

  • produit du DQL
  • reste lisible
  • est fortement recommandé pour débuter

9. Ce que vous devez maîtriser à ce stade

Vous devez être capable de :

  • expliquer ce qu’est une Entity
  • comprendre le rôle d’un Repository
  • savoir à quoi sert l’EntityManager
  • comprendre d’où viennent les données
  • lire une requête DQL simple
  • utiliser Doctrine via l’injection de services

10. Ce que vous n’êtes pas encore censé maîtriser

  • jointures complexes
  • relations avancées (OneToMany, ManyToMany)
  • transactions complexes
  • optimisation SQL

Ces points viendront plus tard.

11. Bonnes pratiques essentielles

  • jamais de SQL dans les contrôleurs
  • les requêtes dans les Repositories
  • logique métier dans des services
  • Doctrine ne gère pas l’affichage

12. À retenir absolument

Doctrine manipule des objets, pas des tables. Les Entities représentent les données. Les Repositories interrogent la base. Le DQL raisonne en objets. Doctrine s’intègre via les services Symfony.

Conclusion

Doctrine est un pilier fondamental de Symfony.

Sans entrer dans la complexité :

  • vous savez maintenant d’où viennent les données
  • comment elles sont représentées
  • comment elles sont récupérées
  • et pourquoi le DQL est essentiel

Ce socle est suffisant pour :

  • lire un projet Symfony
  • écrire des fonctionnalités simples
  • poursuivre vers des usages plus avancés
