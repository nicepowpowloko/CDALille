LE CONTRÔLEUR SYMFONY

Comprendre et maîtriser make:controller

1. Le rôle du contrôleur dans Symfony

1.1 Le contrôleur dans l’architecture MVC

Dans Symfony, le contrôleur est le point d’entrée de votre application côté serveur.

Son rôle est de :

  • recevoir une requête HTTP
  • coordonner le traitement de cette requête
  • appeler les composants nécessaires (services, accès aux données)
  • retourner une réponse HTTP

Le contrôleur n’est pas :

  • un modèle
  • un accès direct à la base de données
  • un conteneur de logique métier complexe

Le contrôleur oriente et orchestre, il ne fait pas le travail métier.

1.2 Le contrôleur Symfony est un service

Un contrôleur Symfony n’est pas un simple fichier PHP exécuté directement.

Il s’agit :

  • d’un service
  • instancié par le container de dépendances
  • déclenché par le système de routage
  • intégré à l’écosystème du framework

Cela implique :

  • une approche orientée objet
  • l’utilisation de l’injection de dépendances
  • le respect des conventions Symfony

2. Que fait la commande make:controller ?

2.1 Objectif de la commande

La commande :

php bin/console make:controller

permet de :

  • créer un contrôleur conforme aux conventions Symfony
  • générer automatiquement une route fonctionnelle
  • créer un template Twig associé

Elle vous évite d’écrire du code répétitif et vous place immédiatement dans une structure correcte.

2.2 Fichiers générés

Après l’exécution de la commande, vous obtenez généralement :

  • un contrôleur dans :
src/Controller/
  • un template Twig dans :
templates/

2.3 Exemple de contrôleur généré

namespace App\Controller;

use Symfony\Bundle\FrameworkBundle\Controller\AbstractController;
use Symfony\Component\HttpFoundation\Response;
use Symfony\Component\Routing\Attribute\Route;

class ArticleController extends AbstractController
{
    #[Route('/article', name: 'article_index')]
    public function index(): Response
    {
        return $this->render('article/index.html.twig');
    }
}

Chaque ligne de ce fichier a une utilité précise et mérite d’être comprise.

3. Comprendre AbstractController

3.1 Pourquoi utiliser AbstractController ?

AbstractController est une classe fournie par Symfony pour faciliter l’écriture des contrôleurs.

Elle vous apporte :

  • des méthodes utilitaires prêtes à l’emploi
  • une meilleure lisibilité du code
  • une réduction du code technique répétitif

Son utilisation est optionnelle, mais parfaitement légitime.

3.2 Fonctionnalités principales fournies

En étendant AbstractController, vous disposez notamment de :

  • rendu de templates Twig :
$this->render()
  • réponses JSON :
$this->json()
  • redirections :
$this->redirectToRoute()
  • gestion des erreurs HTTP :
$this->createNotFoundException()
  • sécurité :
$this->getUser()
$this->isGranted()
  • formulaires :
$this->createForm()

Ces méthodes vous permettent de vous concentrer sur la logique de votre application.

3.3 Ce que fait AbstractController en interne

AbstractController :

  • déclare explicitement les services dont il a besoin
  • permet à Symfony de les injecter automatiquement
  • respecte le principe d’injection de dépendances

Il ne s’agit pas de magie, mais d’une convention maîtrisée.

4. Ce que vous devez maîtriser à ce stade

4.1 Compétences attendues

Lorsque vous créez vos premiers contrôleurs Symfony, vous devez être capable de :

  • générer un contrôleur avec make:controller
  • comprendre le lien entre une route et une méthode
  • retourner une réponse HTTP appropriée
  • afficher une vue avec Twig ou retourner du JSON
  • structurer correctement votre code

L’utilisation d’AbstractController est parfaitement adaptée à ce niveau.

4.2 Responsabilités du contrôleur

Votre contrôleur doit :

  • rester court et lisible
  • déléguer la logique métier à des services
  • utiliser les repositories pour l’accès aux données
  • se limiter à l’orchestration

Un bon contrôleur est souvent simple et peu spectaculaire. C’est un signe de qualité.

5. Comprendre que AbstractController est optionnel

5.1 Savoir ce que signifie “s’en passer”

Il est possible d’écrire un contrôleur sans étendre AbstractController.

Dans ce cas, vous devez :

  • injecter explicitement les services nécessaires
  • créer manuellement les réponses HTTP
  • gérer vous-même le rendu des vues

Exemple :

class ArticleController
{
    public function __construct(
        private Environment $twig
    ) {}

    public function index(): Response
    {
        return new Response(
            $this->twig->render('article/index.html.twig')
        );
    }
}

5.2 Ce que cela implique pour vous

À ce stade, il n’est pas exigé que vous écriviez vos contrôleurs de cette manière.

En revanche, vous devez :

  • savoir que cette possibilité existe
  • comprendre pourquoi AbstractController est utilisé par défaut
  • être capable d’expliquer la différence

La maîtrise complète de cette approche relève d’un approfondissement ultérieur.

6. Bonnes pratiques essentielles

6.1 À faire dans un contrôleur Symfony

  • recevoir et analyser la requête
  • valider les données
  • appeler les bons services
  • retourner une réponse cohérente

6.2 À éviter absolument

  • écrire des requêtes SQL dans le contrôleur
  • placer de la logique métier complexe
  • utiliser directement le container
  • transformer le contrôleur en script procédural

7. Message essentiel à retenir

AbstractController est un outil normal et professionnel. Vous devez comprendre ce qu’il vous apporte. Vous devez savoir qu’il est possible de s’en passer. Vous n’êtes pas tenu de le faire à ce stade.

8. Conclusion

Lorsque vous exécutez :

php bin/console make:controller

vous ne créez pas simplement un fichier PHP.

Vous vous inscrivez dans :

  • une architecture MVC claire
  • une logique orientée objet
  • une organisation professionnelle du code

En maîtrisant le rôle du contrôleur et l’usage d’AbstractController, vous posez des bases solides pour la suite de votre apprentissage de Symfony.
