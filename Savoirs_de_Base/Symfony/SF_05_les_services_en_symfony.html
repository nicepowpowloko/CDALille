LES SERVICES EN SYMFONY

Comprendre, créer et utiliser un service

(dans le framework Symfony)

1. Pourquoi Symfony parle autant de “services” ?

Symfony est un framework orienté architecture. Au cœur de cette architecture se trouve une idée simple :

Toute fonctionnalité réutilisable doit être un service.

Un service est un objet PHP :

  • instancié par Symfony
  • partagé dans l’application
  • réutilisable
  • injecté là où on en a besoin

2. Définition simple d’un service

Un service Symfony est :

  • une classe PHP
  • dont l’instance est gérée par le container
  • et utilisée par d’autres parties de l’application (contrôleurs, autres services…)

Un service fait le travail. Le contrôleur orchestre.

3. Pourquoi ne pas tout mettre dans les contrôleurs ?

Sans services, on obtient souvent ceci :

class OrderController
{
    public function create()
    {
        // validation
        // calcul du total
        // logique métier
        // enregistrement en base
        // envoi d’email
    }
}

Problèmes :

  • contrôleur trop long
  • logique difficile à relire
  • code non réutilisable
  • tests compliqués

Symfony propose une autre approche :

  • le contrôleur délègue
  • le service travaille

4. Exemple simple : un service métier

4.1 Création d’un service

namespace App\Service;

class PriceCalculator
{
    public function calculate(float $price, float $taxRate): float
    {
        return $price * (1 + $taxRate);
    }
}

Cette classe :

  • ne dépend pas de Symfony
  • ne retourne pas de Response
  • contient uniquement de la logique métier

C’est un excellent service.

5. Où placer un service ?

Par convention, les services sont placés dans :

src/Service/

Mais techniquement :

  • un service peut être n’importe où
  • tant qu’il est déclaré ou auto-détecté

6. Le container de services

Symfony utilise un container de dépendances.

Son rôle est de :

  • créer les objets
  • gérer leurs dépendances
  • les fournir aux classes qui en ont besoin

Vous ne créez pas vos services avec new Symfony s’en charge pour vous

7. L’injection de dépendances (principe fondamental)

7.1 Mauvaise pratique (à éviter)

$calculator = new PriceCalculator();

Pourquoi c’est mal ?

  • vous contournez le container
  • vous perdez les avantages de Symfony
  • le code devient rigide

7.2 Bonne pratique : injection par constructeur

class OrderController extends AbstractController
{
    public function __construct(
        private PriceCalculator $calculator
    ) {}

    public function create(): Response
    {
        $total = $this->calculator->calculate(100, 0.2);
        // ...
    }
}

Symfony :

  • voit la dépendance
  • crée le service
  • l’injecte automatiquement

8. Autowiring et autoconfiguration

Symfony moderne utilise deux mécanismes clés :

8.1 Autowiring

Symfony devine quoi injecter grâce au type :

PriceCalculator $calculator

Pas besoin de configuration manuelle.

8.2 Autoconfiguration

Symfony applique automatiquement :

  • les bons comportements
  • les bonnes interfaces
  • les bons tags

Dans la majorité des cas, vous n’avez rien à configurer.

9. services.yaml : faut-il encore l’utiliser ?

Par défaut, Symfony contient :

# config/services.yaml
services:
    App\:
        resource: '../src/'
        autowire: true
        autoconfigure: true

Cela signifie :

  • toutes les classes dans src/ peuvent devenir des services
  • si elles sont utilisées quelque part

Vous n’avez rien à ajouter pour un service simple.

10. Services et contrôleurs : qui fait quoi ?

10.1 Le contrôleur

  • reçoit la requête
  • appelle les services
  • retourne une réponse

10.2 Le service

  • contient la logique métier
  • effectue les calculs
  • applique les règles
  • ne connaît pas HTTP

Un service ne doit pas retourner une Response.

11. Un contrôleur bien écrit

class OrderController extends AbstractController
{
    public function __construct(
        private OrderService $orderService
    ) {}

    public function create(): Response
    {
        $this->orderService->createOrder();
        return new Response('Commande créée');
    }
}

Le contrôleur est :

  • court
  • lisible
  • facile à maintenir

12. Un service peut dépendre d’un autre service

Exemple :

class OrderService
{
    public function __construct(
        private PriceCalculator $calculator
    ) {}
}

Symfony gère toute la chaîne automatiquement.

13. Ce que vous devez maîtriser à ce stade

Vous devez être capable de :

  • expliquer ce qu’est un service
  • créer un service simple
  • l’injecter dans un contrôleur
  • comprendre le rôle du container
  • distinguer logique métier et orchestration

14. Ce qui n’est pas attendu immédiatement

  • écrire des services ultra-génériques
  • manipuler le container directement
  • configurer manuellement tous les services
  • optimiser prématurément

Ces points relèvent d’un approfondissement.

15. Bonnes pratiques essentielles

  • un service = une responsabilité claire
  • pas de logique HTTP dans un service
  • pas de new pour créer des services
  • injection par constructeur
  • services testables indépendamment

16. Message clé à retenir

Un service est une classe métier gérée par Symfony. Le contrôleur orchestre, le service travaille. L’injection de dépendances est centrale. Le container est votre allié, pas un détail technique.

Conclusion

Les services sont le cœur de Symfony. Les comprendre vous permet :

  • d’écrire un code propre
  • de séparer correctement les responsabilités
  • de faire évoluer vos projets sans les casser

Maîtriser les services, c’est passer du “code qui fonctionne” à une application professionnelle.
