CRUD COMPLET EN SYMFONY

Synthèse : Doctrine, Formulaires, Sécurité et Twig

(dans le framework Symfony)

1. Pourquoi un chapitre CRUD de synthèse ?

Vous avez étudié séparément :

  • Doctrine (les données)
  • les Formulaires
  • la Sécurité
  • Twig
  • les Contrôleurs et le Routing

Dans une application réelle, tout cela fonctionne ensemble.

Ce chapitre a pour objectif de :

  • montrer comment ces briques s’articulent
  • comprendre la structure complète d’un CRUD Symfony
  • savoir analyser, écrire ou adapter un CRUD existant

L’objectif n’est pas d’introduire de nouveaux concepts, mais de mettre en cohérence tout ce que vous avez appris.

2. Qu’est-ce qu’un CRUD ?

CRUD est l’acronyme de :

Action Signification
Create Créer
Read Lire
Update Modifier
Delete Supprimer

Un CRUD Symfony standard repose sur :

  • une Entity Doctrine
  • un Repository
  • un FormType
  • un Controller
  • des vues Twig
  • des règles de sécurité

Chaque élément a une responsabilité précise.

3. Génération automatique avec make:crud

Symfony fournit une commande permettant de générer automatiquement un CRUD à partir d’une entité existante : 

php bin/console make:crud

Cette commande :

  • s’appuie sur une Entity Doctrine
  • génère une structure CRUD complète
  • permet d’obtenir rapidement une base fonctionnelle

3.1 Ce que génère réellement make:crud

Pour une entité Article, Symfony génère généralement : 

src/Controller/ArticleController.php
src/Form/ArticleType.php
templates/article/
    index.html.twig
    new.html.twig
    edit.html.twig
    show.html.twig

Tous les composants que vous avez étudiés séparément.

3.2 Limites du code généré

Le CRUD généré :

  • est volontairement générique
  • ne contient pas de logique métier
  • applique peu ou pas de règles de sécurité
  • nécessite presque toujours des adaptations

make:crud est un outil d’accélération, pas un substitut à la compréhension.

3.3 Bonne pratique pédagogique

Vous devez être capable de comprendre, modifier ou réécrire entièrement le code généré par make:crud.

Un développeur qui ne comprend pas ce qui est généré :

  • dépend trop de l’outil
  • produit du code fragile
  • ne maîtrise pas l’architecture

4. La base : l’Entity Doctrine

Exemple : Article

namespace App\Entity;

use Doctrine\ORM\Mapping as ORM;

#[ORM\Entity]
class Article
{
    #[ORM\Id]
    #[ORM\GeneratedValue]
    #[ORM\Column]
    private ?int $id = null;

    #[ORM\Column(length: 255)]
    private string $title;

    public function getId(): ?int
    {
        return $this->id;
    }

    public function getTitle(): string
    {
        return $this->title;
    }

    public function setTitle(string $title): self
    {
        $this->title = $title;
        return $this;
    }
}

L’Entity représente les données. Elle ne gère ni formulaire, ni affichage, ni sécurité.

6. Le FormType : création et modification

namespace App\Form;

use App\Entity\Article;
use Symfony\Component\Form\AbstractType;
use Symfony\Component\Form\FormBuilderInterface;
use Symfony\Component\OptionsResolver\OptionsResolver;

class ArticleType extends AbstractType
{
    public function buildForm(FormBuilderInterface $builder, array $options): void
    {
        $builder->add('title');
    }

    public function configureOptions(OptionsResolver $resolver): void
    {
        $resolver->setDefaults([
            'data_class' => Article::class,
        ]);
    }
}

Le FormType :

  • décrit les champs
  • hydrate l’Entity
  • déclenche la validation

7. Le contrôleur CRUD

7.1 Lister (READ)

#[Route('/article', name: 'article_index')]
public function index(ArticleRepository $repository): Response
{
    return $this->render('article/index.html.twig', [
        'articles' => $repository->findAll(),
    ]);
}

7.2 Créer (CREATE)

#[Route('/article/new', name: 'article_new')]
public function new(
    Request $request,
    EntityManagerInterface $em
): Response {
    $this->denyAccessUnlessGranted('ROLE_USER');

    $article = new Article();
    $form = $this->createForm(ArticleType::class, $article);
    $form->handleRequest($request);

    if ($form->isSubmitted() && $form->isValid()) {
        $em->persist($article);
        $em->flush();

        return $this->redirectToRoute('article_index');
    }

    return $this->render('article/new.html.twig', [
        'form' => $form,
    ]);
}

Sécurité Formulaire Doctrine Redirection

7.3 Modifier (UPDATE)

#[Route('/article/{id}/edit', name: 'article_edit')]
public function edit(
    Article $article,
    Request $request,
    EntityManagerInterface $em
): Response {
    $this->denyAccessUnlessGranted('ROLE_USER');

    $form = $this->createForm(ArticleType::class, $article);
    $form->handleRequest($request);

    if ($form->isSubmitted() && $form->isValid()) {
        $em->flush();
        return $this->redirectToRoute('article_index');
    }

    return $this->render('article/edit.html.twig', [
        'form' => $form,
    ]);
}

Pas besoin de persist() Doctrine suit déjà l’objet

7.4 Supprimer (DELETE)

#[Route('/article/{id}/delete', name: 'article_delete', methods: ['POST'])]
public function delete(
    Article $article,
    EntityManagerInterface $em
): Response {
    $this->denyAccessUnlessGranted('ROLE_ADMIN');

    $em->remove($article);
    $em->flush();

    return $this->redirectToRoute('article_index');
}

Sécurité renforcée Méthode POST Action explicite
