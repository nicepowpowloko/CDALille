LE ROUTING EN SYMFONY

Comprendre les routes et les attributs #[Route]

1. Pourquoi le routing est indispensable

Dans une application Symfony, aucun contrôleur n’est appelé directement.

Lorsqu’un utilisateur accède à une URL, par exemple :

https://example.com/article/12

Symfony doit répondre à une question essentielle :

Quel contrôleur et quelle méthode doivent traiter cette requête ?

Le routing est le mécanisme qui permet de faire ce lien.

2. Le rôle du routing

Le routing sert à :

  • associer une URL à une méthode de contrôleur
  • transmettre des paramètres depuis l’URL
  • identifier une route par un nom
  • centraliser la logique d’accès aux pages

Sans routing :

  • Symfony ne sait pas quoi exécuter
  • les contrôleurs ne sont jamais appelés

Le routing est le pont entre le web (URL) et le code PHP (contrôleurs).

3. Le principe général

Une route définit :

  • un chemin (URL)
  • un contrôleur
  • une méthode
  • éventuellement des paramètres
  • un nom de route

Schéma mental simple :

URL → Route → Contrôleur → Méthode → Response

4. Les routes avec attributs #[Route]

Symfony moderne utilise principalement les attributs PHP (depuis PHP 8).

Exemple simple :

use Symfony\Component\Routing\Attribute\Route;

class HomeController extends AbstractController
{
    #[Route('/', name: 'home')]
    public function index(): Response
    {
        return new Response('Accueil');
    }
}

Ici :

  • #[Route] définit la route
  • / est le chemin
  • home est le nom de la route
  • index() est la méthode appelée

5. Lien entre route et contrôleur

Symfony fonctionne ainsi :

  1. Une requête HTTP arrive (/)
  2. Symfony analyse toutes les routes connues
  3. Il trouve une route qui correspond
  4. Il appelle la méthode du contrôleur associée
  5. La méthode retourne une Response

Le contrôleur n’est jamais appelé par vous, mais par Symfony.

6. Les paramètres d’URL

Les routes peuvent contenir des paramètres dynamiques.

Exemple :

#[Route('/article/{id}', name: 'article_show')]
public function show(int $id): Response
{
    return new Response('Article ' . $id);
}

Si l’URL est :

/article/42

Alors :

  • {id} vaut 42
  • Symfony injecte automatiquement la valeur dans la méthode

6.1 Typage des paramètres

Symfony respecte les types PHP :

public function show(int $id)

Si la valeur ne correspond pas :

  • Symfony retourne une erreur 404

Le typage est une sécurité implicite.

7. Contraintes sur les paramètres

Vous pouvez limiter les valeurs acceptées :

#[Route('/article/{id}', name: 'article_show', requirements: ['id' => '\d+'])]

Ici :

  • {id} doit être un nombre

Sinon, la route ne correspond pas.

8. Les noms de routes

Chaque route doit avoir un nom unique :

#[Route('/contact', name: 'contact')]

Les noms de routes servent à :

  • générer des URLs
  • éviter d’écrire des chemins en dur
  • faciliter la maintenance

Exemple :

$this->redirectToRoute('contact');

Si le chemin change, le nom reste le même.

On ne travaille jamais avec des URLs en dur, mais avec des noms de routes.

9. Générer des URLs à partir des routes

9.1 Redirection

return $this->redirectToRoute('article_show', [
    'id' => 12
]);

Symfony génère automatiquement :

/article/12

9.2 Dans Twig

<a href="{{ path('article_show', { id: 12 }) }}">
    Voir l’article
</a>

10. Une route = une action

Bonne pratique essentielle :

  • une route appelle une seule méthode
  • une méthode répond à une seule intention

!!! À éviter :

#[Route('/article')]
public function article()
{
    if ($_GET['action'] === 'delete') {
        // ...
    }
}

Chaque action doit avoir sa propre route.

11. Méthodes HTTP (GET, POST…)

Par défaut, une route accepte toutes les méthodes HTTP.

Vous pouvez restreindre :

#[Route('/article', name: 'article_create', methods: ['POST'])]

Cela permet de :

  • respecter le fonctionnement HTTP
  • sécuriser les accès
  • clarifier l’intention

12. Où sont définies les routes ?

Avec les attributs :

  • directement au-dessus des méthodes
  • dans les contrôleurs

Symfony détecte automatiquement les routes dans :

src/Controller/

13. Ce que vous devez maîtriser à ce stade

Vous devez être capable de :

  • expliquer le rôle du routing
  • définir une route avec #[Route]
  • relier une URL à un contrôleur
  • utiliser des paramètres d’URL
  • nommer correctement une route
  • générer une URL à partir de son nom

14. À retenir absolument

Le routing relie le web au code. Une URL n’appelle jamais un fichier PHP. Une route appelle une méthode de contrôleur. Les noms de routes sont plus importants que les chemins.

Conclusion

Maintenant que vous savez créer des contrôleurs, il est indispensable de comprendre comment Symfony décide de les appeler.

Le routing est :

  • central
  • structurant
  • indispensable à toute application Symfony

Une bonne compréhension du routing vous permettra :

  • d’écrire des contrôleurs propres
  • de naviguer correctement dans l’application
  • d’éviter les erreurs classiques de débutant
