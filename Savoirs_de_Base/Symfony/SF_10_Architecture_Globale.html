ARCHITECTURE GLOBALE D’UNE APPLICATION SYMFONY

Comprendre le fonctionnement d’ensemble

(dans le framework Symfony)

1. Pourquoi parler d’architecture globale ?

Après avoir vu :

  • les contrôleurs
  • le routing
  • la Request et la Response
  • les vues Twig
  • les services
  • Doctrine

Il devient indispensable de prendre du recul.

Ce chapitre a pour but de répondre à une question simple mais fondamentale :

Que se passe-t-il réellement lorsqu’un utilisateur accède à une page Symfony ?

2. Qui appelle qui ? (vue d’ensemble)

Dans Symfony, rien n’est appelé au hasard.

Voici la chaîne logique simplifiée :

Navigateur
   ↓
Request HTTP
   ↓
Front Controller (index.php)
   ↓
Router
   ↓
Contrôleur
   ↓
Services / Doctrine
   ↓
Twig (optionnel)
   ↓
Response
   ↓
Navigateur

Chaque élément a un rôle précis. Aucun ne fait le travail d’un autre.

3. Le cycle d’une requête HTTP pas à pas

3.1 Arrivée de la requête

Un utilisateur accède à une URL :

https://example.com/article/12

Le navigateur envoie une Request HTTP.

Symfony reçoit toutes les requêtes via un seul fichier : 

public/index.php

C’est le Front Controller.

3.2 Le routing

Symfony analyse la Request et demande au routeur :

“Existe-t-il une route correspondant à cette URL ?”

Si une route est trouvée :

  • elle désigne un contrôleur
  • et une méthode

Sinon :

  • Symfony retourne une erreur 404

3.3 L’appel du contrôleur

Symfony :

  • instancie le contrôleur via le container
  • injecte les dépendances nécessaires
  • appelle la méthode correspondante

Vous n’appelez jamais un contrôleur vous-même.

3.4 Le travail métier

Le contrôleur :

  • lit la Request
  • délègue le travail aux services
  • récupère des données via Doctrine
  • prépare les données pour l’affichage

Le contrôleur orchestre, il ne calcule pas.

3.5 La génération de la Response

Selon le contexte :

  • Twig génère du HTML
  • ou Symfony renvoie du JSON
  • ou une redirection
  • ou une erreur HTTP

Le contrôleur retourne toujours une Response.

Symfony l’envoie ensuite au navigateur.

4. Le rôle central du container de services

Le container est le cœur technique de Symfony.

Il est responsable de :

  • créer les objets
  • gérer les dépendances
  • injecter les services
  • garantir une architecture cohérente

Sans container :

  • pas de services
  • pas d’injection
  • pas d’architecture propre

Vous ne travaillez jamais directement avec le container :

  • vous déclarez vos dépendances
  • Symfony fait le reste

5. Où mettre quoi ? (organisation du code)

C’est une question clé pour éviter le désordre.

5.1 Le contrôleur

src/Controller/

Rôle :

  • recevoir la Request
  • appeler les services
  • retourner une Response

Il ne doit pas :

  • contenir de logique métier
  • accéder directement à la base
  • produire du HTML complexe

5.2 Les services

src/Service/ (par convention)

Rôle :

  • contenir la logique métier
  • effectuer les calculs
  • appliquer les règles fonctionnelles

Ils ne doivent pas :

  • connaître HTTP
  • retourner des Response
  • dépendre de Twig

5.3 Les entités Doctrine

src/Entity/

Rôle :

  • représenter les données
  • définir la structure métier

Elles ne doivent pas :

  • contenir de logique applicative complexe
  • accéder à la base
  • dépendre de Symfony

5.4 Les repositories

src/Repository/

Rôle :

  • interroger la base de données
  • centraliser les requêtes
  • utiliser DQL / QueryBuilder

Ils ne doivent pas :

  • faire de l’affichage
  • contenir de logique métier transverse

5.5 Les vues Twig

templates/

Rôle :

  • afficher les données
  • produire du HTML

Elles ne doivent pas :

  • contenir du PHP
  • faire des calculs
  • accéder aux données

6. Séparation des responsabilités (principe fondamental)

Symfony repose sur une idée simple :

Chaque couche a une responsabilité claire.

Élément Responsabilité
Controller Orchestration
Service Logique métier
Repository Accès aux données
Entity Modèle métier
Twig Affichage
Routing Navigation

Quand cette règle est respectée, l’application reste lisible et maintenable.

7. Ce que vous devez retenir à ce stade

Vous devez être capable de :

  • expliquer le cycle d’une requête HTTP
  • comprendre qui appelle qui
  • identifier le rôle du container
  • savoir où placer votre code
  • lire l’architecture d’un projet Symfony existant

8. Ce que vous n’êtes pas encore censé maîtriser

À ce stade, il n’est pas attendu que vous maîtrisiez :

  • les événements Symfony
  • les listeners / subscribers
  • la compilation du container
  • les optimisations avancées

Ces sujets feront l’objet d’un approfondissement ultérieur.

9. Message clé de synthèse

Symfony est une orchestration d’objets, pas une suite de scripts. Le framework appelle votre code, jamais l’inverse. Une bonne architecture repose sur la séparation des responsabilités.

Conclusion

Ce chapitre marque la fin du socle fondamental Symfony.

Vous avez maintenant une vision claire :

  • du fonctionnement global
  • des rôles de chaque composant
  • de la place de votre code dans l’application

Cette compréhension est essentielle pour :

  • continuer sereinement
  • éviter les erreurs structurelles
  • aborder les approfondissements à venir avec de bonnes bases
