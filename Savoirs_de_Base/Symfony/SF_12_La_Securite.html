LA SÉCURITÉ EN SYMFONY

Authentification, utilisateurs et gestion des rôles

(dans le framework Symfony)

1. Pourquoi la sécurité est un sujet central

Toute application un minimum réaliste doit répondre à ces questions :

  • qui est l’utilisateur ?
  • est-il authentifié ?
  • a-t-il le droit d’accéder à cette page ou à cette action ?

Symfony fournit un système de sécurité structuré, intégré au framework, qui évite :

  • les bricolages maison
  • les mots de passe stockés en clair
  • les contrôles dispersés dans le code

La sécurité n’est pas optionnelle, elle est transversale.

2. Les grands principes de la sécurité Symfony

La sécurité Symfony repose sur quatre concepts fondamentaux :

  1. User (l’utilisateur)
  2. Authentification (se connecter)
  3. Rôles / autorisations
  4. Contrôle d’accès

Ces éléments sont pensés pour fonctionner ensemble.

3. L’utilisateur (User)

3.1 Le rôle de la classe User

Un utilisateur est représenté par une Entity qui implémente une interface Symfony. 

use Symfony\Component\Security\Core\User\UserInterface;

class User implements UserInterface
{
    // ...
}

Cette classe représente :

  • l’identité de l’utilisateur
  • ses rôles
  • ses informations d’authentification

L’utilisateur est un objet métier, stocké en base de données.

3.2 Les éléments essentiels d’un User

Un utilisateur doit au minimum fournir :

  • un identifiant (email, username…)
  • un mot de passe hashé
  • une liste de rôles

Exemple simplifié :

private string $email;
private string $password;
private array $roles = [];

4. L’authentification

4.1 Qu’est-ce que l’authentification ?

L’authentification consiste à répondre à la question :

“Est-ce bien cette personne qui tente d’accéder à l’application ?”

Symfony gère cela via :

  • un formulaire de connexion
  • un système d’authenticator
  • une comparaison de mot de passe hashé

Vous ne comparez jamais les mots de passe vous-même.

4.2 Le formulaire de connexion

Symfony fournit des outils pour générer :

  • un formulaire de login
  • un traitement automatique
  • une gestion des erreurs

Le contrôleur de login :

  • affiche le formulaire
  • ne traite pas manuellement les identifiants

5. Le mot de passe : un point critique

Symfony ne stocke jamais un mot de passe en clair.

  • les mots de passe sont hashés
  • Symfony utilise des algorithmes sécurisés
  • la vérification est automatique

Exemple (au niveau conceptuel) :

password_hash()
password_verify()

Vous ne devez jamais implémenter votre propre système de hash.

6. Les rôles

6.1 Qu’est-ce qu’un rôle ?

Un rôle est une autorisation abstraite.

Exemples courants :

  • ROLE_USER
  • ROLE_ADMIN

Les rôles sont :

  • stockés dans l’utilisateur
  • utilisés pour contrôler l’accès

6.2 Hiérarchie des rôles (principe)

Un rôle peut en inclure un autre :

ROLE_ADMIN
  └─ ROLE_USER

Un administrateur est aussi un utilisateur.

7. Vérifier les droits dans un contrôleur

Symfony fournit des méthodes simples.

7.1 Vérification explicite

$this->denyAccessUnlessGranted('ROLE_ADMIN');

Si l’utilisateur n’a pas le rôle :

  • Symfony renvoie une erreur 403

7.2 Vérification conditionnelle

if ($this->isGranted('ROLE_ADMIN')) {
    // action autorisée
}

8. Protéger des routes

La sécurité peut être appliquée avant même l’appel du contrôleur.

Exemples de règles :

  • certaines pages accessibles uniquement aux utilisateurs connectés
  • certaines pages réservées aux administrateurs

Cela permet d’avoir des contrôleurs plus simples et plus sûrs.

9. L’utilisateur connecté dans l’application

Dans un contrôleur, vous pouvez accéder à l’utilisateur courant :

$user = $this->getUser();

Cela permet :

  • d’adapter l’affichage
  • de filtrer des données
  • de personnaliser l’expérience

Dans Twig :

{% if app.user %}
    Bonjour {{ app.user.email }}
{% endif %}

10. Sécurité et services

La sécurité s’intègre naturellement avec les services.

Exemple :

  • un service métier peut vérifier les droits
  • un contrôleur délègue les décisions complexes

La logique de sécurité complexe ne doit pas être dispersée.

11. Bonnes pratiques essentielles

  • ne jamais stocker de mot de passe en clair
  • ne jamais gérer l’authentification “à la main”
  • centraliser les règles de sécurité
  • utiliser les rôles, pas des booléens
  • séparer authentification et logique métier

12. Ce que vous devez maîtriser à ce stade

Vous devez être capable de :

  • expliquer ce qu’est un utilisateur Symfony
  • comprendre le principe d’authentification
  • savoir à quoi servent les rôles
  • protéger un contrôleur ou une route
  • utiliser l’utilisateur connecté

13. Ce que vous n’êtes pas encore censé maîtriser

  • voters personnalisés
  • firewall multiples complexes
  • OAuth, JWT, SSO
  • sécurité avancée des API

Ces points feront l’objet d’un approfondissement ultérieur.

14. À retenir absolument

La sécurité est intégrée à Symfony. L’authentification identifie l’utilisateur. Les rôles définissent les droits. Le framework applique les règles avant votre code.

Conclusion

La sécurité Symfony permet de :

  • protéger votre application
  • structurer les accès
  • éviter les erreurs critiques
  • écrire un code professionnel et maintenable

Avec cette base, vous pouvez désormais :

  • créer des zones protégées
  • gérer des utilisateurs
  • préparer des projets réalistes
