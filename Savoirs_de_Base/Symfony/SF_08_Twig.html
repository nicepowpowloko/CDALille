TWIG - LES VUES EN SYMFONY

Séparer l’affichage de la logique applicative

(dans le framework Symfony)

1. Pourquoi utiliser Twig ?

Dans une application Symfony, le contrôleur :

  • reçoit la requête
  • prépare les données
  • ne doit pas produire du HTML directement

Le rôle de l’affichage est confié à Twig, le moteur de templates de Symfony.

Twig permet de :

  • écrire du HTML lisible
  • insérer des données dynamiques
  • éviter le PHP dans les vues
  • respecter la séparation des responsabilités

Le contrôleur orchestre, Twig affiche.

2. Le rôle d’une vue

Une vue est un fichier :

  • dont le rôle est uniquement l’affichage
  • sans logique métier
  • sans accès à la base de données
  • sans traitement complexe

Dans Symfony, les vues sont généralement placées dans :

templates/

3. Rendre une vue avec render()

Dans un contrôleur, l’affichage se fait avec :

return $this->render('home/index.html.twig');

Cette méthode :

  • utilise Twig
  • génère du HTML
  • retourne une Response

4. Passer des variables à une vue

Le contrôleur peut transmettre des données à Twig.

4.1 Depuis le contrôleur

return $this->render('home/index.html.twig', [
    'title' => 'Accueil',
    'username' => 'Manuel'
]);

4.2 Dans la vue Twig

<h1>{{ title }}</h1>
<p>Bienvenue {{ username }}</p>

{{ }} permet d’afficher une variable.

Twig échappe automatiquement les données (sécurité XSS).

5. La syntaxe de base de Twig

5.1 Affichage

{{ variable }}

5.2 Conditions

{% if user %}
    <p>Bonjour {{ user.name }}</p>
{% else %}
    <p>Bonjour invité</p>
{% endif %}

5.3 Boucles

<ul>
{% for article in articles %}
    <li>{{ article.title }}</li>
{% endfor %}
</ul>

Twig reste volontairement simple.

6. Séparation HTML / PHP

Avec Twig :

  • pas de <?php ?> dans les vues
  • pas de logique métier
  • pas de SQL
  • pas de traitements complexes

!!! À éviter absolument dans une vue :

  • calculs
  • accès à la base de données
  • conditions métier complexes

Tout cela appartient aux services ou aux contrôleurs.

7. L’héritage de templates

Twig permet de créer une structure commune à toutes les pages.

7.1 Le template de base

{# templates/base.html.twig #}
<!DOCTYPE html>
<html>
<head>
    <title>{% block title %}Mon site{% endblock %}</title>
</head>
<body>

<header>
    <h1>Mon application</h1>
</header>

<main>
    {% block body %}{% endblock %}
</main>

</body>
</html>

7.2 Étendre un template

{# templates/home/index.html.twig #}
{% extends 'base.html.twig' %}

{% block title %}Accueil{% endblock %}

{% block body %}
    <p>Bienvenue sur le site</p>
{% endblock %}

L’héritage permet :

  • une mise en page cohérente
  • une maintenance simplifiée
  • un code plus propre

8. Générer des liens avec les routes

Twig s’intègre avec le routing :

<a href="{{ path('home') }}">Accueil</a>

Si la route change :

  • le lien reste valide
  • aucune URL en dur

9. Organisation recommandée des vues

Exemple :

templates/
├── base.html.twig
├── home/
│   └── index.html.twig
├── article/
│   ├── index.html.twig
│   └── show.html.twig

Cette organisation reflète :

  • les contrôleurs
  • les routes
  • les intentions

10. Ce que vous devez maîtriser à ce stade

Vous devez être capable de :

  • expliquer le rôle de Twig
  • afficher une vue avec render()
  • passer des variables à Twig
  • utiliser conditions et boucles simples
  • comprendre l’héritage de templates
  • respecter la séparation HTML / PHP

11. À retenir absolument

Twig sert uniquement à l’affichage. Le contrôleur prépare les données. Le HTML est séparé du PHP. L’héritage de templates structure l’application.

Conclusion

Twig est un outil simple mais puissant. Il vous permet de :

  • produire du HTML propre
  • maintenir une architecture claire
  • éviter les mélanges dangereux entre logique et affichage

Maîtriser Twig, c’est écrire des vues propres, lisibles et professionnelles.
