LA REQUEST ET LA RESPONSE EN SYMFONY

Comprendre les échanges HTTP entre le client et le serveur

(dans le framework Symfony)

1. Pourquoi parler de Request et de Response ?

Le web repose sur un principe fondamental :

Un client envoie une requête (Request), le serveur renvoie une réponse (Response).

Symfony ne fait pas exception. Il se contente de structurer proprement ce mécanisme.

Comprendre Request et Response, c’est :

  • comprendre comment les données arrivent dans votre application
  • comprendre comment vous répondez au navigateur ou à une API
  • comprendre le cœur du fonctionnement web

2. Le cycle simplifié d’une requête HTTP

Voici ce qu’il se passe lorsqu’un utilisateur accède à une page :

  1. Le navigateur envoie une Request
  2. Symfony reçoit cette Request
  3. Le routing choisit un contrôleur
  4. Le contrôleur traite la Request
  5. Le contrôleur retourne une Response
  6. Symfony envoie la Response au client

Tout votre code s’inscrit entre la Request et la Response.

3. La classe Request

La classe Request représente la requête HTTP entrante.

Elle contient :

  • l’URL
  • les paramètres
  • les données envoyées par un formulaire
  • les headers
  • la méthode HTTP (GET, POST…)

3.1 Obtenir la Request dans un contrôleur

Symfony injecte automatiquement la Request :

use Symfony\Component\HttpFoundation\Request;

public function index(Request $request): Response
{
    // ...
}

Vous ne créez jamais une Request vous-même.

4. Les différentes sources de données dans la Request

4.1 query - paramètres GET

Correspond aux paramètres dans l’URL :

/search?q=symfony&page=2

Récupération :

$q = $request->query->get('q');
$page = $request->query->get('page');

query = GET

4.2 request - données POST

Correspond aux données envoyées par un formulaire ou un POST classique. 

$name = $request->request->get('name');

request = POST

4.3 attributes - paramètres de route

Ce sont les paramètres définis dans les routes :

#[Route('/article/{id}', name: 'article_show')]
public function show(Request $request): Response
{
    $id = $request->attributes->get('id');
}

Ces paramètres proviennent du routing, pas du formulaire.

!!! En pratique, Symfony les injecte souvent directement dans la méthode : 

public function show(int $id): Response

Mais il est important de comprendre d’où ils viennent.

5. Résumé des sources de données

Source Origine Exemple
query URL ?page=2
request Formulaire / POST name=Jean
attributes Route /article/{id}

6. La classe Response

La classe Response représente ce que le serveur renvoie au client.

Une Response contient :

  • un contenu
  • un code HTTP
  • des headers

Exemple simple :

use Symfony\Component\HttpFoundation\Response;

return new Response('Bienvenue');

7. Les codes HTTP essentiels à connaître

Les codes HTTP indiquent le résultat de la requête.

7.1 Codes courants

Code Signification
200 OK
201 Créé
302 Redirection
400 Mauvaise requête
403 Accès interdit
404 Ressource introuvable
500 Erreur serveur

Exemple :

return new Response('Introuvable', 404);

Symfony fournit aussi des méthodes dédiées :

throw $this->createNotFoundException();

8. Différence entre réponse HTML et réponse JSON

8.1 Réponse HTML

Utilisée pour :

  • afficher des pages
  • renvoyer des vues Twig
return $this->render('home/index.html.twig');

Le navigateur reçoit du HTML.

8.2 Réponse JSON

Utilisée pour :

  • API
  • AJAX
  • front JavaScript
return $this->json([
    'status' => 'ok',
    'data' => $data
]);

Le navigateur ou le client reçoit du JSON.

9. Comment choisir entre HTML et JSON ?

Cela dépend :

  • du type de client (navigateur, application JS, mobile)
  • de l’architecture de votre application
Cas Type de réponse
Site classique HTML
API REST JSON
Application hybride Les deux

Symfony permet de gérer les deux très simplement.

10. Un contrôleur bien écrit avec Request et Response

#[Route('/search', name: 'search')]
public function search(Request $request): Response
{
    $term = $request->query->get('q');

    return new Response('Recherche : ' . $term);
}

Ce contrôleur :

  • lit la Request
  • traite les données
  • retourne une Response claire

11. Ce que vous devez maîtriser à ce stade

Vous devez être capable de :

  • expliquer le rôle de Request et Response
  • récupérer des données GET, POST et de route
  • comprendre les codes HTTP
  • retourner une réponse HTML ou JSON
  • lire et écrire un contrôleur orienté HTTP

12. À retenir absolument

La Request représente ce que le client envoie. La Response représente ce que le serveur renvoie. Les paramètres viennent de plusieurs sources. Les codes HTTP sont essentiels à la communication web.

Conclusion

La maîtrise de Request et Response est fondamentale :

  • pour comprendre le web
  • pour comprendre Symfony
  • pour écrire des applications robustes

Symfony ne cache pas le HTTP. Il vous aide à l’utiliser correctement.
