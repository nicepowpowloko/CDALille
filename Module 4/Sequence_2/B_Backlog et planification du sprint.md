# Fiche B - Backlog et planification du sprint (GitHub Issues/Projects)

## Idée générale

Le backlog est la liste des choses à faire, triée par importance. Il évolue tout le temps. Plus un élément est proche d’être développé, plus il doit être précis. Pour éviter le flou, on transforme les besoins en tickets clairs, puis on planifie un sprint avec un objectif unique.

## User story et critères d’acceptation

Une user story décrit un besoin du point de vue d’un utilisateur, avec une valeur. Elle doit être testable. Les critères d’acceptation indiquent ce qui doit être vrai pour dire “c’est validé”. Sans critères, on confond vite “fait” et “presque fait”.

## GitHub Issues et Project

Sur GitHub, une story devient une Issue. Le Project sert de tableau de suivi (Backlog, Ready, In progress, Review, Done). On cherche à limiter le travail “en cours” pour éviter que tout reste à moitié fini.

## Planification sprint

Planifier un sprint, c’est choisir une petite quantité de travail faisable, alignée sur un Sprint Goal. On privilégie des tickets petits. Si un ticket paraît trop gros, on le découpe avant de le prendre.

### Encadré vocabulaire

* **Backlog**: liste ordonnée de tout ce qu’on pourrait faire.
* **Issue**: ticket GitHub qui décrit un travail concret.
* **Critères d’acceptation**: conditions vérifiables de validation.
* **Sprint Goal**: phrase qui décrit ce qu’on veut atteindre dans le sprint.

### Mini exemple Cassandre

Backlog MVP:

1. “on peut créer un compte”
2. “on peut se connecter”
3. “on voit son profil”
   On planifie un sprint avec l’objectif: “à la fin du sprint, on peut se connecter et accéder à son espace”. On met les Issues dans “Ready”, puis on les fait passer vers “In progress”, “Review”, “Done”.
